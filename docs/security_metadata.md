# 종목 메타데이터 자동 분류와 캐시

## 목적

새 보유종목을 추가했을 때 총괄현황의 데스크톱 히트맵이 종목을 가능한 한 자동으로 섹터에 배치합니다. 사용자가 섹터를 직접 입력하지 않아도 시장과 티커를 기준으로 종목 메타데이터를 조회하고, 결과를 계정 포트폴리오에 함께 저장합니다.

## 조회 순서

1. 보유 데이터에 이미 저장된 수동 섹터
2. 현재 주요 보유종목에 대한 내장 분류
3. 계정 포트폴리오에 저장된 최근 메타데이터
4. 앱 프로세스의 공용 메모리 캐시
5. yfinance `Ticker.get_info()`의 Yahoo Finance 메타데이터
6. 조회하거나 분류할 수 없으면 `기타`

미국 주식은 티커를 그대로 조회합니다. 국내 주식은 6자리 종목코드에 `.KS`를 붙여 먼저 조회하고, 정보가 없으면 `.KQ`로 다시 조회합니다.

## 분류 기준

조회 결과의 `sectorKey`, `industryKey`, `quoteType`을 함께 사용합니다.

- 바이오·헬스케어
- 반도체·전자
- 귀금속·광업
- 조선·산업재
- 정보기술, 금융, 소재, 산업재
- 경기소비재, 필수소비재, 커뮤니케이션
- 에너지, 부동산, 유틸리티
- ETF·펀드, 지수·파생, 가상자산
- 기타

세부 산업 정보가 있으면 광범위한 섹터보다 우선합니다. 예를 들어 `technology` 섹터 안에서도 `semiconductors`는 `반도체·전자`로 분류합니다.

## 캐시와 과부하 방지

- 성공한 메타데이터는 각 사용자의 기존 포트폴리오 JSON 안에 저장하고 30일간 재사용합니다.
- 같은 앱 프로세스에서는 성공 결과를 7일간 공용 메모리 캐시로 재사용합니다.
- 실패 결과는 24시간 동안 재시도하지 않고, 24시간이 지나면 다시 조회할 수 있습니다.
- 앱 실행 한 번에 새 네트워크 조회는 최대 2종목으로 제한합니다.
- 종목당 조회 제한 시간은 4초입니다.

여러 신규 종목을 한 번에 추가하면 처음 두 종목부터 분류되고, 이후 화면 실행에서 다음 종목이 순차적으로 처리됩니다. 저장된 결과는 다음 로그인부터 바로 재사용합니다.

## 저장 구조와 보안

다음 필드가 기존 `holdings` 행에 선택적으로 추가됩니다.

- `sector`, `sector_key`
- `industry`, `industry_key`
- `quote_type`
- `metadata_source`, `metadata_fetched_at`, `metadata_error`

별도 Supabase 테이블이나 migration은 필요하지 않습니다. 기존 `portfolio_snapshots.payload_json`에 사용자 보유 데이터와 함께 저장되므로 기존 Supabase Auth와 RLS 경계를 그대로 따릅니다. 공개 앱에 service role key를 추가하지 않습니다.

메모리 캐시에는 시장, 티커, 공개 종목 메타데이터만 저장하며 사용자 ID, 수량, 평균단가 같은 개인 정보는 저장하지 않습니다.

## 한계

Yahoo Finance가 종목 메타데이터를 제공하지 않거나 잘못 제공하면 `기타`로 표시될 수 있습니다. 메타데이터는 화면 그룹화를 위한 참고 정보이며 투자 판단이나 공식 업종 분류를 의미하지 않습니다.
